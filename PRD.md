# Seam: Cloudflare-Native Sync Engine

> `@iamkaf/seam` is a typed mutation and sync protocol for structured applications on Cloudflare Workers and D1.

## Thesis

Seam syncs versioned records with server-authoritative sequencing. It is not a document database, a CRDT framework, an event-sourcing system, or an ORM. Every write is an app-defined mutation that runs through validation, authorization, optimistic concurrency checks, and a durable commit before clients can observe it.

The core promise is simple:

1. A mutation changes a bounded write set of sync-visible records, usually one record.
2. Every record change and its replication-log entry commit atomically.
3. Clients keep one cursor for their active scopes and pull coalesced record changes after that cursor.
4. Optimistic client state is always replaced by the server response or by a later pull.

This model is for workflow-heavy apps, dashboards, back offices, CLIs, moderation queues, and machine clients where clear ownership, predictable conflicts, and inspectable state matter more than automatic merging.

## 1. Product Boundaries

### What Seam Is

Seam is a small sync substrate:

- Typed record envelopes persisted in D1.
- App-owned mutation handlers.
- Bounded write sets for domain operations that must update a few records together.
- Import helpers for large machine-authored snapshots.
- Scope-based authorization hooks.
- A global replication cursor.
- Pull-based sync with optional realtime invalidation.
- React and non-React client libraries.

### What Seam Is Not

| Not this | Reason |
| --- | --- |
| CRDT framework | Seam chooses explicit conflicts over automatic merges. |
| Event store | Current state is stored as records; events are a side channel. |
| ORM | Apps own read models and named queries. |
| Offline-first database | V1 supports short offline queues, not long-lived divergent replicas. |
| Text collaboration engine | No OT, presence, or character-level merge. |
| Auth provider | Apps own identity, sessions, tokens, and tenant provisioning. |

### Design Drivers

Seam v1 is shaped by three reference use cases:

- **Ordered collaboration apps** need scoped records, optimistic drag and drop, durable activity, and template-based setup.
- **Catalog and moderation apps** need project-scoped records, public and private read rules, contributor suggestions, moderator approval that updates more than one record, and machine imports that can touch hundreds or thousands of records.
- **Apps with external identity providers** need Seam to accept resolved actors from the host app while leaving roles, token scopes, resource permissions, and moderation policy app-owned.

These requirements imply two rules: normal UI mutations should stay small and record-oriented, while imports and setup flows need a separate bulk path that logs every changed record without pretending each import is a user gesture.

## 2. Data Model

### Records

Seam syncs records: typed, versioned, JSON-serializable entities with stable IDs.

```typescript
interface SeamRecord<T = Record<string, unknown>> {
  id: string;
  type: string;
  version: number;
  data: T;
  scopeKind: string;
  scopeId: string;
  createdAt: string;
  createdBy: string;
  updatedAt: string;
  updatedBy: string;
  deletedAt?: string;
}
```

Record IDs are globally unique ULIDs generated by the server. The record ID alone identifies a record; there is no `(type, id)` composite identity.

`data` is opaque to Seam's sync machinery. Apps register schemas and mutation handlers that give `data` its meaning. Seam owns the envelope fields, versioning, scoping, and replication mechanics.

### Scopes

Every record belongs to exactly one scope.

```typescript
interface SeamScope {
  kind: string;
  id: string;
}
```

Scopes are the unit of permission checks, WebSocket subscription, bootstrap, and pull filtering. Records do not move between scopes in v1. Moving an entity across scopes is modeled as a delete in the old scope and a create in the new scope.

### Versions

Versions are monotonic per record. Creates start at version `1`; every update or soft delete increments the version by one.

V1 has one write-concurrency mode: strict optimistic concurrency control. Every update/delete write must provide `expectedVersion`. If any stored version differs, the whole write set is rejected with `VERSION_CONFLICT` and the current authorized record that caused the conflict.

Last-writer-wins writes are deliberately not part of v1. Admin overwrite behavior can be built as app code that fetches the latest record and retries with an explicit version.

### Sequence Log

The sequence log is a global append-only invalidation index.

```typescript
interface SeqEntry {
  seq: number;
  opId: string;
  scopeKind: string;
  scopeId: string;
  recordType: string;
  recordId: string;
  mutationType: string;
  actorId: string;
  timestamp: string;
  version: number;
}
```

The log does not store historical record snapshots. Pull scans the log to discover which records changed, then fetches the current version of those records. If a record changed three times while a client was offline, the client receives the current record once and advances past all three sequence entries.

The log is for replication, not product activity feeds, audit trails, billing events, or webhooks. Durable app events use the outbox table described later.

## 3. D1 Schema

```sql
CREATE TABLE records (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  scope_kind TEXT NOT NULL,
  scope_id TEXT NOT NULL,
  version INTEGER NOT NULL DEFAULT 1,
  data TEXT NOT NULL,
  created_at TEXT NOT NULL,
  created_by TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  updated_by TEXT NOT NULL,
  deleted_at TEXT,
  last_op_id TEXT NOT NULL
) STRICT;

CREATE INDEX idx_records_scope_type_id
  ON records(scope_kind, scope_id, type, id);

CREATE INDEX idx_records_active_list
  ON records(scope_kind, scope_id, type, id)
  WHERE deleted_at IS NULL;

CREATE TABLE seq_log (
  seq INTEGER PRIMARY KEY AUTOINCREMENT,
  op_id TEXT NOT NULL,
  scope_kind TEXT NOT NULL,
  scope_id TEXT NOT NULL,
  record_type TEXT NOT NULL,
  record_id TEXT NOT NULL,
  mutation_type TEXT NOT NULL,
  actor_id TEXT NOT NULL,
  timestamp TEXT NOT NULL,
  version INTEGER NOT NULL,
  hint TEXT
) STRICT;

CREATE INDEX idx_seq_scope
  ON seq_log(scope_kind, scope_id, seq);

CREATE INDEX idx_seq_op
  ON seq_log(op_id, seq);

CREATE UNIQUE INDEX idx_seq_op_record
  ON seq_log(op_id, record_id);

CREATE TABLE seam_batch_assertions (
  op_id TEXT NOT NULL,
  step TEXT NOT NULL,
  ok INTEGER NOT NULL CHECK (ok = 1),
  PRIMARY KEY(op_id, step)
) STRICT;

CREATE TABLE mutation_receipts (
  actor_id TEXT NOT NULL,
  client_mutation_id TEXT NOT NULL,
  request_hash TEXT NOT NULL,
  seq INTEGER NOT NULL,
  scope_kind TEXT NOT NULL,
  scope_id TEXT NOT NULL,
  response_json TEXT NOT NULL,
  created_at TEXT NOT NULL,
  PRIMARY KEY(actor_id, client_mutation_id)
) STRICT;

CREATE TABLE outbox (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  op_id TEXT NOT NULL,
  seq INTEGER NOT NULL,
  scope_kind TEXT NOT NULL,
  scope_id TEXT NOT NULL,
  record_type TEXT,
  record_id TEXT,
  event_type TEXT NOT NULL,
  payload TEXT NOT NULL,
  actor_id TEXT NOT NULL,
  created_at TEXT NOT NULL
) STRICT;

CREATE INDEX idx_outbox_seq ON outbox(seq);
```

Notes:

- `last_op_id` is an internal guard used to make conditional commits safe in D1 batches.
- `seam_batch_assertions` is an internal rollback trigger. Failed compare-and-swap checks insert `ok = 0`, violating the check constraint and rolling back the whole batch.
- `seq_log.hint` is optional JSON for realtime invalidation hints. It is not a durable event payload.
- `mutation_receipts.response_json` stores the exact successful mutation response for idempotent replay.
- `outbox` stores durable app events that must be processed at least once.

## 4. Server API

### Definition Surface

```typescript
import { createSeamServer, defineRecordType, SeamError } from '@iamkaf/seam/server';

const Card = defineRecordType('card', {
  schema: z.object({
    title: z.string(),
    listId: z.string(),
    position: z.number(),
    assigneeId: z.string().optional(),
  }),
});

const seam = createSeamServer({
  records: [Card],
  resolveContext: async (request, env) => {
    const session = await verifySession(request, env);
    if (session) return { actorId: session.userId, actorType: 'user' };
    throw new SeamError('UNAUTHORIZED');
  },
  mutations: {
    'card.create': {
      input: z.object({ boardId: z.string(), listId: z.string(), title: z.string() }),
      scope: (input) => ({ kind: 'board', id: input.boardId }),
      authorize: (ctx) => ctx.hasRole('editor'),
      execute: async (input) => ({
        type: 'card',
        data: { title: input.title, listId: input.listId, position: 0 },
      }),
      events: (_input, result) => [
        { type: 'card.created', payload: { cardId: result.id } },
      ],
    },
    'card.move': {
      record: 'card',
      input: z.object({ listId: z.string(), position: z.number() }),
      authorize: (ctx) => ctx.hasRole('editor'),
      validate: (input, { current }) => {
        if (current.deletedAt) throw new SeamError('NOT_FOUND');
        if (!input.listId) throw new SeamError('INVALID', 'listId required');
      },
      execute: async (input, { current }) => ({
        ...current.data,
        listId: input.listId,
        position: input.position,
      }),
      events: (input, result) => [
        { type: 'card.moved', payload: { cardId: result.id, listId: input.listId } },
      ],
    },
  },
});

export default { fetch: seam.fetch };
```

### Mutate Wire Protocol

Framework fields are separate from app input.

```typescript
interface MutateRequest {
  mutation: string;
  input: Record<string, unknown>;
  id?: string;
  expectedVersion?: number;
  clientMutationId: string;
}

interface MutateSuccess {
  ok: true;
  records: SeamRecord[];
  record?: SeamRecord;
  seq: number;
  clientMutationId: string;
}

interface MutateFailure {
  ok: false;
  error: {
    code:
      | 'UNAUTHORIZED'
      | 'FORBIDDEN'
      | 'NOT_FOUND'
      | 'TYPE_MISMATCH'
      | 'VERSION_CONFLICT'
      | 'INVALID'
      | 'IDEMPOTENCY_KEY_REUSED';
    message: string;
    record?: SeamRecord;
  };
  clientMutationId: string;
}
```

Every mutation request must include `clientMutationId`. Record-targeting mutations require `id` and `expectedVersion`. Create mutations omit both; the server generates record IDs. `seq` is the highest sequence assigned by the operation. `record` is the primary changed record for ergonomic single-record mutations; `records` contains every sync-visible record changed by the operation.

The idempotency request hash is computed from the canonical full mutation envelope: `mutation`, `input`, `id`, and `expectedVersion`. It is scoped by `(actorId, clientMutationId)`. Reusing the same key with a different envelope returns `IDEMPOTENCY_KEY_REUSED`.

### Mutation Pipeline

1. Resolve actor context.
2. Parse the wire envelope and validate app input.
3. Load any receipt for `(actorId, clientMutationId)`.
4. If a matching receipt exists and its request hash differs, return `IDEMPOTENCY_KEY_REUSED`.
5. If a matching receipt exists and the actor still has access to the receipt scope, return the stored response. If access was revoked, return `NOT_FOUND`.
6. For create mutations, resolve scope from input.
7. For record-targeting mutations, load the current primary record by ID. If missing, return `NOT_FOUND`.
8. Authorize against the resolved scope. Unauthorized access to existing records returns `NOT_FOUND` to avoid existence leaks. Unauthorized creates return `FORBIDDEN`.
9. For record-targeting mutations, verify `current.type` matches the mutation's declared record type.
10. For record-targeting mutations, compare `expectedVersion` with `current.version`.
11. Run app validation.
12. Run app execution to produce a write set. Any additional update/delete writes must include their own loaded current record and `expectedVersion`.
13. Commit the write set, seq entries, idempotency receipt, commit effects, and outbox events in one guarded D1 batch.
14. Best-effort broadcast an invalidation message to the scope's Durable Object.

## 5. Write Sets

Most mutations change one record. Some real app operations need a few records to change together: accepting a translation suggestion updates the suggestion and the approved translation, while updating a card may also update a workspace summary record. Seam supports bounded write sets for those cases.

```typescript
type SeamWrite =
  | {
      op: 'create';
      type: string;
      scope: SeamScope;
      data: Record<string, unknown>;
      id?: string;
    }
  | {
      op: 'update' | 'delete';
      id: string;
      type: string;
      expectedVersion: number;
      data?: Record<string, unknown>;
    };

interface MutationCommit {
  writes: SeamWrite[];
  primaryRecordId?: string;
  events?: Array<{ type: string; payload: Record<string, unknown>; recordId?: string }>;
  effects?: CommitEffect[];
}
```

Rules:

- UI mutations should usually emit one write. Multi-record write sets are for domain invariants, not convenience.
- V1 caps interactive write sets at 25 records by default. Apps can lower the cap.
- Each changed record gets its own `seq_log` entry with the same `op_id`.
- All records in one write set must belong to one scope in v1.
- Every update/delete write uses strict `expectedVersion`.
- The response includes all changed records, ordered with the primary record first when there is one.

### Commit Effects

Commit effects are app-owned SQL statements included in the same batch as the write set. They exist for indexes, uniqueness reservations, and small strongly-consistent projections that the app needs immediately.

Commit effect rules:

- They may only write app-owned tables, never Seam core tables.
- They must be derived from the validated mutation and loaded current records.
- They run inside the same D1 batch and rollback with the write set.
- They are not sync-visible and do not create extra `seq_log` entries.
- Large projections, notifications, exports, search, and webhooks should use the durable outbox instead.

Examples:

- A collaboration app can reserve or update a workspace code lookup table.
- A collaboration app can maintain a compact workspace summary table for the home screen.
- A catalog app can maintain an export lookup table for approved translations.

### Bulk Imports And Seeds

Large setup and import flows are not normal interactive mutations. Seam provides server-side import helpers for machine/admin paths:

```typescript
await seam.importRecords({
  actorId: 'token:catalog-importer',
  scope: { kind: 'project', id: projectId },
  mutationType: 'catalog.import',
  records,
  chunkSize: 500,
});
```

Import semantics:

- Each created, updated, or deleted record still gets a `seq_log` entry.
- Each chunk commits atomically; very large imports are not one D1 transaction.
- Imports are server-authoritative and do not use client `expectedVersion`; the importer computes creates, updates, and deletes from the primary immediately before each chunk.
- Imports return an `importId`, changed counts, and the highest committed seq.
- Apps that need coherent all-at-once publication should use an app-level publish marker record, such as `catalogVersion`, and only flip that marker in the final chunk.
- Imports can write outbox events for indexing, export rebuilds, and audit trails.

This covers template-based workspace creation, source catalog imports, approved translation imports, test fixtures, and migrations without making the public mutation API an unbounded batch endpoint.

### Ordered Collections

Apps should model ordered UI state so common reorder actions mutate one record.

For Kanban-style lists and cards:

- Store `position` on the ordered record.
- Use rank strings or sparse fractional positions so moving one card only updates that card.
- Rebalance positions with a background/import operation when ranks become dense.
- Avoid full-list reorder mutations for drag and drop.

This lets a card move remain a single-record mutation even when the UI shows a whole column reorder.

### Translation Catalogs

Translation/catalog apps should model translation state as records plus app-owned projections:

- `project` records hold project metadata and visibility.
- `sourceString` records hold canonical source text, placeholder signature, active state, and import revision.
- `translation` records hold approved text for one source string and locale.
- `suggestion` records hold contributor proposals and moderation status.
- Moderator approval is a small write set: update the suggestion and create/update the approved translation.
- Machine imports use `importRecords`; coherent publication uses a `catalogVersion` or `localeVersion` marker record.
- Export endpoints are named queries over Seam records or app-maintained projection tables, not generic Seam filters.

This keeps contributor UI reactive without forcing build/export endpoints through the sync API.

## 6. Atomic Commit Protocol

Seam relies on D1 batch transactions: if any statement in the batch errors, the whole batch rolls back. A conditional `UPDATE` that affects zero rows is not an error, so Seam turns failed write preconditions into constraint errors using `seam_batch_assertions`.

The server generates one `opId` for the whole write set. Each changed record stores that `opId` in `last_op_id`; each `seq_log` row for the operation uses the same `op_id`.

### Create Write

```sql
INSERT INTO records (
  id, type, scope_kind, scope_id, version, data,
  created_at, created_by, updated_at, updated_by, deleted_at, last_op_id
) VALUES (?, ?, ?, ?, 1, ?, ?, ?, ?, ?, NULL, ?);

INSERT INTO seq_log (
  op_id, scope_kind, scope_id, record_type, record_id,
  mutation_type, actor_id, timestamp, version, hint
)
SELECT ?, scope_kind, scope_id, type, id, ?, ?, ?, version, ?
FROM records
WHERE id = ? AND last_op_id = ?;
```

### Update Or Delete Write

```sql
UPDATE records
SET data = ?,
    version = version + 1,
    updated_at = ?,
    updated_by = ?,
    deleted_at = ?,
    last_op_id = ?
WHERE id = ? AND type = ? AND version = ?;

INSERT INTO seam_batch_assertions (op_id, step, ok)
VALUES (?, ?, CASE changes() WHEN 1 THEN 1 ELSE 0 END);

INSERT INTO seq_log (
  op_id, scope_kind, scope_id, record_type, record_id,
  mutation_type, actor_id, timestamp, version, hint
)
SELECT ?, scope_kind, scope_id, type, id, ?, ?, ?, version, ?
FROM records
WHERE id = ? AND last_op_id = ?;
```

The assertion row is deleted later in the same batch after all writes, receipts, effects, and outbox rows have been inserted:

```sql
DELETE FROM seam_batch_assertions WHERE op_id = ?;
```

### Receipt, Effects, And Outbox

After all record writes and `seq_log` inserts, the batch inserts one receipt. The stored response contains every changed record and the highest `seq` for the operation.

```sql
INSERT INTO mutation_receipts (
  actor_id, client_mutation_id, request_hash, seq,
  scope_kind, scope_id, response_json, created_at
)
SELECT ?, ?, ?, MAX(seq), ?, ?, ?, ?
FROM seq_log
WHERE op_id = ?;
```

Outbox inserts are operation-level. They may optionally point at a primary record.

```sql
INSERT INTO outbox (
  op_id, seq, scope_kind, scope_id, record_type, record_id,
  event_type, payload, actor_id, created_at
)
SELECT ?, MAX(seq), ?, ?, ?, ?, ?, ?, ?, ?
FROM seq_log
WHERE op_id = ?;
```

Commit effects run in the same batch between the record writes and receipt insert. If a commit effect fails, D1 rolls back the whole operation.

If the receipt insert fails because another request committed the same `clientMutationId`, the whole batch rolls back. The server reloads the receipt and returns it if the request hash matches.

After the batch, the server reads `seq_log` by `op_id` from the primary. If the expected number of seq rows exists, the operation committed. If the batch failed because an assertion inserted `ok = 0`, the server reloads current records and returns `VERSION_CONFLICT` or `NOT_FOUND` as appropriate.

This protocol preserves the core invariant: every visible record change has a matching `seq_log` entry, receipt, and any produced outbox rows from the same batch.

## 7. Reads And Sync

Sync endpoints read from the D1 primary in v1. Replica reads can return a `seq_log` row without the corresponding current record version, which would let a client advance past an update it never received. Named app queries may use replicas if the app accepts eventual consistency.

### Pull Request

```typescript
interface PullRequest {
  afterSeq: number;
  scopes: SeamScope[];
  limit?: number;
  untilSeq?: number;
}

interface PullResponse {
  records: SeamRecord[];
  seq: number;
  hasMore: boolean;
  revokedScopes?: SeamScope[];
}
```

`afterSeq` is the client's coverage cursor for its active scopes. `untilSeq` is optional and is used when joining new scopes. `limit` defaults to 500 and caps at 1000. The limit applies to scanned `seq_log` entries, not returned records.

### Pull Algorithm

1. Check each requested scope. Revoked scopes are returned in `revokedScopes` and omitted from the scan.
2. Query `seq_log` once per authorized scope using `(scope_kind, scope_id, seq)`.
3. Apply `seq > afterSeq` and, if present, `seq <= untilSeq`.
4. Merge entries by global `seq` and take up to `limit` entries.
5. Deduplicate by `record_id`, keeping the highest scanned version per record.
6. Fetch the current record rows from the primary.
7. Verify each fetched row has `version >= highest scanned version` for that record. If not, do not advance past the first unmaterializable seq.
8. Return current records, including soft-deleted records.
9. Set `seq` to the highest covered sequence.

For normal pulls without `untilSeq`, the highest covered sequence is the highest scanned relevant seq. If no relevant entries exist, `seq` remains `afterSeq`.

For bounded pulls with `untilSeq`, if all relevant entries through `untilSeq` fit in the response, `seq` is `untilSeq` even when no records changed. This lets the client prove that existing scopes are caught up to a bootstrap watermark.

### Bootstrap

Bootstrap snapshots one or more scopes from the primary and returns the current global sequence watermark.

```typescript
interface BootstrapRequest {
  scopes: SeamScope[];
}

interface BootstrapResponse {
  records: SeamRecord[];
  seq: number;
  revokedScopes?: SeamScope[];
}
```

The snapshot includes soft-deleted records so clients can evict stale cached rows.

### Adding Scopes

The client must not add a new scope to the pull filter until existing scopes are covered through the bootstrap watermark.

```typescript
async function addScopes(newScopes: SeamScope[]) {
  const snapshot = await client.bootstrap({ scopes: newScopes });
  const watermark = snapshot.seq;
  const acceptedScopes = withoutRevoked(newScopes, snapshot.revokedScopes ?? []);

  while (store.activeScopes.length > 0 && store.cursor < watermark) {
    const delta = await client.pull({
      afterSeq: store.cursor,
      scopes: store.activeScopes,
      untilSeq: watermark,
    });
    store.mergeAll(delta.records);
    store.evictScopes(delta.revokedScopes ?? []);
    store.advanceCursor(delta.seq);
    if (!delta.hasMore) break;
  }

  store.mergeAll(snapshot.records);
  store.markActive(acceptedScopes);
  store.advanceCursor(watermark);
}
```

If the cursor is expired during this process, the client bootstraps all active scopes plus the new scopes and replaces its local cache for those scopes.

### Listing Records

Built-in listing is intentionally small.

```typescript
const cards = await client.list({
  type: 'card',
  scope: { kind: 'board', id: boardId },
  limit: 50,
  after: lastId,
});
```

The server authorizes the scope and runs keyset pagination by record ID:

```sql
SELECT * FROM records
WHERE scope_kind = ?
  AND scope_id = ?
  AND type = ?
  AND deleted_at IS NULL
  AND id > ?
ORDER BY id
LIMIT ?;
```

No generic JSON filtering endpoint exists. Apps define named query endpoints and indexes for richer reads.

## 8. Authorization Model

Apps own authentication and authorization. Seam asks the app to resolve an actor and then calls app-owned authorization functions for the target scopes and mutations.

Seam should integrate with external identity systems through generic adapter hooks, not built-in provider-specific code. An adapter answers "who is this actor?" and may return response headers that the host app must forward, such as session-rotation `Set-Cookie` headers. The app still decides local roles, token scopes, public/private access, and moderation policy.

```typescript
interface SeamContext {
  actorId: string | null;
  actorType: 'anonymous' | 'user' | 'token' | 'service';
  tokenScopes?: string[];
  globalRoles?: string[];
  scopeRoles: Map<string, string>;
  hasRole(role: string): boolean;
  hasGlobalRole(role: string): boolean;
  isOwner(record: SeamRecord): boolean;
}
```

`hasRole` checks the role in the current target scope only. Roles are flat strings in v1; `admin` is not automatically a superset of `editor`. Apps define role semantics in `authorize` functions.

Seam does not require a permissions table. It can ship an optional default scope-role table for simple apps, but production apps may resolve roles from app tables:

- A collaboration app can resolve roles from membership tables and token resource restrictions.
- A catalog app can resolve public reads from project visibility, moderator roles from app user tables, and machine imports from token scopes.
- An app backed by centralized identity can resolve browser identity through that provider and then enrich the Seam context locally.

Read behavior:

- `getById` loads the record, resolves its scope, authorizes, and returns `NOT_FOUND` if unauthorized.
- `list` authorizes the requested scope and returns `NOT_FOUND` if unauthorized.
- `bootstrap` and `pull` may allow anonymous access for public scopes if the app authorizer permits it.
- `pull` returns revoked scopes in `revokedScopes` so the client can evict cached data and stop requesting them.

Mutation behavior:

- Existing-record mutations return `NOT_FOUND` when the actor lacks access to the record's scope.
- Create mutations return `FORBIDDEN` when the actor lacks access to the target scope.
- Mutations require a non-anonymous actor. Public anonymous access is read-only in v1.
- `TYPE_MISMATCH` and `VERSION_CONFLICT` are only returned after authorization succeeds.

## 9. Client Architecture

### Store

```typescript
interface ClientStore {
  records: Map<string, SeamRecord>;
  cursor: number;
  activeScopes: SeamScope[];
  pending: MutationQueue;
  subscriptions: Map<string, Set<Callback>>;
}
```

The v1 store is in memory. Optional `localStorage` can persist last-known records across reloads, but pending mutations are not durable in v1. A page refresh loses pending mutations.

### Projections

Apps rarely render raw records. The client should expose projection helpers that derive app-specific view models from the local store.

```typescript
const workspace = useSeamProjection({
  scope: { kind: 'board', id: boardId },
  select: (records) => createWorkspaceProjection(records),
});
```

Projection rules:

- Projections run over authoritative records plus optimistic pending records.
- Deleted records are excluded by default but can be included for tombstone-aware projections.
- Selectors should be pure and synchronous.
- Seam tracks record subscriptions by scope and type; apps own sorting, filtering, joining, and view-specific shaping.

Collaboration apps can project records into board/workspace view models. Catalog apps can project source strings, translations, and suggestions into project workbench rows while keeping export endpoints as server-side named queries.

### Optimistic Mutations

1. Capture a base snapshot of the target record.
2. Add the mutation to the pending queue.
3. Apply optimistic records to the store.
4. Send the mutation with `expectedVersion` and `clientMutationId`.
5. On success, replace optimistic state with every authoritative record in the response and remove the pending entry.
6. On `VERSION_CONFLICT`, roll back the failed optimistic change, merge the server record, and abort later pending mutations that depended on the failed optimistic state.
7. Mutations targeting different records proceed independently.

Optimistic creates use a temporary client ID. The server response contains the real ULID, and `clientMutationId` lets the client replace the temporary record.

### Sync Loop

- Poll every 30 seconds by default.
- If WebSocket invalidation is connected, pause polling except for auth revalidation.
- On WebSocket disconnect, resume polling with backoff: 1s, 2s, 4s, 8s, capped at the configured poll interval.
- On each pull, merge records, evict `revokedScopes`, and advance the cursor only to the response `seq`.
- If `hasMore` is true, immediately pull again.

## 10. Realtime

Realtime is an optimization over pull.

```typescript
interface InvalidationMessage {
  type: 'invalidate';
  seq: number;
  hint?: { recordType: string; recordId: string; mutationType: string };
}
```

Each scope may have a Durable Object that accepts authorized WebSocket connections. After commit, the Worker best-effort sends an invalidation to that scope's DO. The DO broadcasts the invalidation; clients then call pull.

The DO never sends authoritative record payloads. It does not re-check per-record field permissions, and it does not provide delivery guarantees. Missed WebSocket messages are recovered by polling from the last cursor.

V1 can ship without WebSocket support. Polling is the correctness path.

## 11. Durable App Events

Mutation handlers may return typed events. Seam writes those events to `outbox` in the same guarded batch as the write set and `seq_log` entries.

Outbox consumers provide durable background processing for search indexing, webhooks, billing, compliance exports, and notifications that must reflect committed mutations.

```typescript
interface OutboxConsumerState {
  consumerName: string;
  lastOutboxId: number;
}
```

A consumer reads `outbox WHERE id > lastOutboxId ORDER BY id LIMIT ?`, performs work idempotently, and advances its cursor after success. Delivery is at least once. Event payloads must include everything the consumer needs; consumers should not depend on the current record still matching the mutation-time state.

Queue enqueue after commit is allowed as a latency optimization, but it is not the durable source of truth. The outbox is.

## 12. Errors

| Code | Meaning |
| --- | --- |
| `UNAUTHORIZED` | No valid actor context. |
| `FORBIDDEN` | Actor is authenticated but cannot create in the target scope. |
| `NOT_FOUND` | Missing record, unauthorized existing record, deleted record, or inaccessible scope. |
| `TYPE_MISMATCH` | Authorized caller targeted an existing record with the wrong mutation type. |
| `VERSION_CONFLICT` | Authorized caller supplied a stale `expectedVersion`. |
| `INVALID` | Request shape or app validation failed. |
| `IDEMPOTENCY_KEY_REUSED` | Same actor/key was used for a different mutation envelope. |
| `CURSOR_EXPIRED` | Pull cursor is older than retained `seq_log`. |

## 13. Retention And Garbage Collection

`seq_log` grows with every committed mutation. Apps must prune it.

Seam will provide `pruneSeqLog(beforeSeq)`, which deletes sequence entries older than a configured threshold and records the minimum retained seq. If a client pulls with `afterSeq` below the minimum retained seq, the server returns `CURSOR_EXPIRED`; the client must bootstrap its active scopes.

Receipts can be pruned with the same retention horizon. After a receipt is pruned, retrying an old `clientMutationId` is treated as a new mutation and may conflict by version.

Soft-deleted records should not be hard-deleted while any retained `seq_log` entry can still reference them. V1 does not include automatic hard-delete GC.

Outbox retention is separate. Apps prune outbox rows only after all registered consumers have advanced past them.

## 14. Consistency Guarantees

Seam v1 guarantees:

- Successful mutation responses reflect the full write set, log entries, receipt, commit effects, and any produced outbox rows committed in the D1 primary.
- Every visible record change has exactly one `seq_log` entry.
- Pull never advances past a scanned seq unless the corresponding current record version is materialized from the primary.
- Bounded multi-record write sets commit atomically within one D1 batch.
- WebSocket invalidation is best effort; polling recovers missed notifications.
- Optimistic state is speculative and always yields to successful mutation responses or pull results.

Seam v1 does not guarantee:

- All-or-nothing transactions for very large imports that exceed one D1 batch.
- Durable client-side pending mutations across page refresh.
- Historical reconstruction of every intermediate record version from `seq_log`.
- Generic server-side filtering over JSON data.
- Proactive WebSocket disconnect on permission revocation.

## 15. Scale Expectations

| Dimension | V1 expectation | Notes |
| --- | --- | --- |
| Records per scope | Fewer than 10,000 | Larger scopes need chunked bootstrap in v2. |
| Active scopes per client | 10 or fewer | Pull runs one indexed query per scope. |
| Interactive write set size | 25 records or fewer | Imports use the bulk helper instead. |
| Mutations per DB | D1 single-writer bound | Keep mutation handlers and batches short. |
| WebSocket connections per DO | About 100 | Shard or multiplex in v2 for higher fan-out. |
| `seq_log` size | App-managed | Prune regularly and force bootstrap for expired cursors. |

A global sequence leaks cross-scope activity through cursor gaps. Apps where activity volume is sensitive should use separate D1 databases per tenant.

## 17. Design Tests

The implementation must ship tests for the invariants, not just happy paths:

- Conditional update failure creates no `seq_log`, receipt, or outbox row.
- Successful create/update/delete creates exactly one `seq_log` row.
- Successful multi-record write set creates one `seq_log` row per changed record and one receipt for the operation.
- Any failed write in a multi-record write set rolls back all record writes, commit effects, seq rows, receipt, and outbox rows.
- Commit effects roll back with the write set and cannot produce additional seq rows.
- Duplicate `clientMutationId` returns the original response and does not create a second record.
- Reusing `clientMutationId` with a different envelope returns `IDEMPOTENCY_KEY_REUSED`.
- Pull coalesces multiple changes to one record and advances past all scanned seq entries.
- Pull with `untilSeq` advances to the watermark even when no active-scope records changed.
- Pull does not advance when a materialized record version is older than the scanned seq entry.
- Bootstrap plus bounded pull does not skip changes in existing scopes or newly-added scopes.
- Import helpers log every changed record and report the highest committed seq.
- Anonymous public reads can bootstrap public scopes, but anonymous mutations are rejected.
- Unauthorized existing-record access returns `NOT_FOUND`, not `TYPE_MISMATCH` or `VERSION_CONFLICT`.
- Revoked scopes in multi-scope pull are returned in `revokedScopes` and do not block authorized scopes.
